# How to Review AI-Generated Python Code Before Merge

If your team uses Copilot, Claude Code, Cursor, or similar tools, the hard part is no longer generating code.

The hard part is **verifying it before it reaches `main`**.

GitHub's current Copilot docs explicitly say to validate Copilot feedback carefully and supplement it with human review. Anthropic's Claude Code security docs say Claude Code uses a permission-based model, but users are still responsible for reviewing proposed code and commands before approval.

That is the boundary.

AI review can assist you. It does not close the verification loop for you.

For Python teams, that means the question is not:

- "Which assistant wrote this?"

It is:

- "How do we prove this change is real, safe, and still enforcing the protections our repo depends on?"

## What verification should prove

Before an AI-generated Python change merges, your workflow should prove four things.

### 1. The code is real

Imports, helper functions, decorators, config keys, and library APIs must actually exist in the repository or dependency set you are running.

This is where hallucinated imports, phantom calls, and stale mocks show up.

### 2. The diff did not remove an existing protection

Broad refactors often look fine because the resulting file is clean. The real bug is a missing decorator, permission branch, middleware registration, or validation path that disappeared during cleanup.

### 3. The change did not add insecure shortcuts

Examples:

- `verify=False`
- `DEBUG=True`
- hardcoded secrets
- unsafe shell execution
- weak crypto

These are classic "make it work quickly" patterns, and AI tools generate them often.

### 4. The repository enforces the same rule every time

If verification depends on whether one reviewer happened to notice a bad diff on a tired Friday afternoon, it is not a workflow. It is luck.

That is why the merge gate matters.

## The Python failure modes to check first

### Hallucinated references

This is the most obvious AI-generated code failure mode and still one of the easiest to miss in review:

```python
def create_order(payload):
    validated = validate_order_schema(payload)
    inventory = reserve_inventory(validated.items)
    payment = charge_customer(validated.total)
    return persist_order(validated, inventory, payment)
```

The orchestration looks reasonable. The problem is that one or more of those helpers may not exist anywhere in the repo.

The same issue shows up in decorators and imports:

```python
from auth_utils import require_workspace_access

@require_workspace_access
def get_report(request):
    ...
```

That code may read like an existing project abstraction even when the decorator was invented by the model.

If you want the narrower deep dive on this class of problem, read [How to Catch Hallucinated Imports and Phantom Calls in AI-Generated Python Code](/use-cases/ai-generated-code-security).

### Regressions hidden inside clean refactors

The more dangerous failure mode is often deletion.

This route starts protected:

```python
@login_required
@rate_limit("10/m")
def export_customer_data(request):
    customer_id = request.GET["customer_id"]
    return generate_export(customer_id)
```

Then the AI refactor lands like this:

```python
def export_customer_data(request):
    customer_id = request.GET["customer_id"]
    return generate_export(customer_id)
```

The function still works. The tests may still pass. But the route is not enforcing the same security model anymore.

This is exactly the class of problem covered in [How to Catch Removed Auth Checks and Security Regressions in AI-Generated PRs](/use-cases/ai-security-regressions-in-prs).

### Insecure convenience code

AI tools are very willing to trade safety for immediacy:

```python
response = requests.get(url, verify=False)

decoded = jwt.decode(token, algorithms=["none"])

app.config["DEBUG"] = True
```

Each example solves a short-term problem:

- certificate trouble
- JWT decoding trouble
- local debugging convenience

Each example is also the sort of thing that survives review when the code around it looks otherwise neat.

### Dead paths and stale test scaffolding

AI-generated diffs also leave behind code that is no longer part of the real execution path:

- helper functions nobody calls
- unfinished stubs
- renamed symbols with stale mocks
- dead cleanup code after an aggressive refactor

These are not always direct vulnerabilities, but they increase review noise, hide intent, and make it easier for real security regressions to blend in.

## A practical verification workflow for Python teams

### 1. Start with a baseline scan

Run the repository once before you judge the new AI-generated change:

```bash
pip install skylos
skylos . -a
```

In the current Skylos workflow, `-a` enables the broader local audit path: dead code, risky flows, secrets, quality checks, and SCA findings.

This is important because AI verification gets messy fast when reviewers cannot tell the difference between:

- old project noise
- newly introduced risk
- and protections that the current diff removed

### 2. Scan the changed lines, not only the whole tree

The safest documented diff-aware command in the Skylos repo is:

```bash
skylos . --diff origin/main --danger --quality
```

This changes the review question from:

> "What problems exist somewhere in this repository?"

to:

> "What risky behavior is present in the code this PR actually changed?"

That is the better question for AI-generated code review because assistants often produce large, plausible refactors. The higher-risk issue is frequently a deleted guardrail or a newly introduced shortcut inside the diff.

### 3. If the repository includes LLM features, verify that surface too

If your Python application includes prompts, tool-calling, RAG, or LLM endpoints, normal code verification is not enough by itself.

Use:

```bash
skylos discover .
skylos defend .
```

In the main Skylos repo, `discover` maps LLM integrations and `defend` checks those integrations for missing guardrails such as prompt injection exposure, unsafe output handling, weak isolation, missing rate limits, or missing cost controls.

That step is only necessary when the repository actually includes an AI feature surface. It is not required for every generic Python service.

### 4. Keep assistant-specific controls, but do not confuse them with verification

Use the vendor-specific controls where they help:

- GitHub Copilot repository custom instructions and review rules
- Claude Code approval prompts and project-scoped MCP approvals
- Cursor approval and MCP configuration boundaries

Those controls shape behavior.

They do not replace repository scanning or PR gating.

For the tool-specific workflows, see:

- [How to Review Claude Code Output for Python Security Regressions](/use-cases/claude-code-security-review)

### 5. Put the merge gate in CI once

After the local workflow feels useful, make it systemic:

```bash
skylos cicd init
```

That gives the team a shared merge barrier instead of a best-effort habit.

If you are already using GitHub Actions, pair this with:

- [Python Security Scanner for GitHub Actions](/use-cases/python-security-github-actions)
- [How to Secure GitHub Actions for Python Repos](/use-cases/secure-github-actions-python)

### Optional: measure where AI-authored risk is showing up

If your team also wants attribution, Skylos includes an AI provenance command:

```bash
skylos provenance . --json
```

That is not the verification step itself. It is the visibility step that helps you answer whether risky changes are clustering around specific AI-assisted workflows.

## The shortest reliable default

If you want the minimum setup that is hard to regret, use this:

```bash
pip install skylos
skylos . -a
skylos . --diff origin/main --danger --quality
skylos cicd init
```

If the repository includes real LLM features, add:

```bash
skylos discover .
skylos defend .
```

That is enough to move the team from "we reviewed what the AI suggested" to "we verified the change before merge."

## Final thought

The most useful thing about the current AI coding tools is that they make code generation cheap.

The most important thing about verification is that it stays expensive unless you systematize it.

The teams that handle AI-generated Python code well are not the teams with the most enthusiastic prompts. They are the teams that make verification boring, repeatable, and hard to skip.
